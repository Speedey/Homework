Searching Algorythmes

The first one consisted in/still consists in cheking every single element in the list, no matter on the size of the list, and looks if it is the element we are looking for. The complexity was exponentiel, it grew with the size of the list. And thats why there is the Second option. This algorythm takes the list, and modifies it to its needs. For example, if its hungry it will consume part of the list. It modifys the list so, it calculates the middle, checks if the element we are looking for is either bigger or smaller than the middle and voila; cuts of the beginning or the end off, since we dont need it. That process is repeated until we have found our element.

Binary and Systems

Binary, oh boy, The language of the PC, 0 and 1, the base of most eclectronics. It truly isnt that hard, to be honest, its like html: looks scary but it aint. Binary, or Base 2(0 and 1) can only be written with 0 and 1, every number has a binary counterpart, which you count by the 0 and 1, for example: 0 (base10)= 0(base2), 1(base10)= 1(base2) - simple right? well yes, but actually no, you can line up these 0 and 1 to make bigger numbers, 15 = 1111. With each new "number slot" representing a power of 2.

PC's usually run with a OS, be it windows, linux, or mac os. They all have there advatages and disadvatages, but these OS's will either be a 32 or 64 bit system, which defines the amount of bits (binary digits) they can work with. Lets keep it simple and stay on a max of 8 Bits. In case you are over 8 bits, there will be an occurence called overflow, that means that all the bits longer than the 8th will be cut off. #Binary numbers are read from right to left, if there is overflow = [(110)10010011], the ones in the parentheses will be cut off. How does one represent positive and negative numbers? well the first(left to right) digit is usually the one wich defines if a number is positive(1) or negative(o)
